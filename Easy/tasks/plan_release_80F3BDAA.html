      Steps
      Definir critério de sucesso da release.

      Existem dois tipos de releases  :

      • end date driven release. O objetivo é estar em produção ou disponível para usuários finais antes de um prazo.
      • feature driven release. A lista de requisitos é conhecida e a release finalizará quando todos os requisitos serão implementados.

      Critérios de sucesso na release dependem do tipo.

      Estimar backlog items

      A estimativa é realizada por item. É melhor trabalhar primeiro nos itens mais importantes.

      A estimativa é realizada pela equipe. A unidade de medida usada para estimativas é comumente o ponto. O intervalo de valores é escolhido a partir dos Números Fibonacci: 1, 2, 3, 5, 8 e 13.

       As técnicas de estimativa são, de preferência, baseadas em uma abordagem de estimação coletiva. O "Planejamento de Poker" deve ser a regra do jogo e também trabalhar com analogias, comparando itens estimados, funciona bem.

      Definir tamanho das sprints

      O comprimento depende do projeto e suas restrições e é definido na reunião da equipe Easy.

      Geralmente entre 15 e 21 dias.
      Estimar velocidade

      Idealmente, a equipe deveria ter trabalhado juntas em um projeto para que sua velocidade real ainda fosse conhecida. Esse valor seria então usado para essa versão, eventualmente ajustado à natureza do projeto. Atenção no entanto, como um ponto não tem necessariamente o mesmo valor em diferentes projetos.

      Se não for o caso, a velocidade deve ser estimada. A melhor maneira é trabalhar no conteúdo do primeiro sprint e descobrir o que a equipe pensa ser capaz de implementar durante este primeiro sprint. Isso dá uma estimativa de velocidade que pode ser usada para o planejamento de release. Então, o planejamento será baseado na velocidade medida em sprints.

      Associar backlog items para as sprints

      Com os seguintes parâmetros:

      • Estimativa de cada item,
      • Priorizar itens,
      • Estimar velocidade para a release,

      Os itens são afetados para as sprints da release.

      É normal ajustar, ou seja, alterar ligeiramente a priorização dos itens, de modo que a velocidade pode ser a mais precisa. Os itens devem ser associados aos primeiros sprints (2 ou 3 primeiros), mas não é necessário fazê-lo para todos os sprints do release em primeiro lugar.

      Key Considerations

      É um trabalho coletivo iniciado pelo Product Owner.
